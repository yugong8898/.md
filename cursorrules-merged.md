# WLYD 项目开发规则

## 一、基础规范

### 1. 语言规则
- 回复使用中文
- 必要时会明确要求英文

### 2. 代码提交规则
- 无明确命令时不做任何代码提交操作
- 只在用户明确要求时才生成 commit 信息

### 3. Commit 信息格式
使用类型前缀 + 简要描述：
- `feat:` - 新功能
- `fix:` - 修复 bug
- `refactor:` - 重构代码
- `style:` - 样式调整
- `docs:` - 文档更新
- `perf:` - 性能优化
- `test:` - 测试相关
- `chore:` - 构建/工具相关

### 4. 原内容保留规则（强制）
- 涉及修改已有内容时，原则上不直接删除原内容。
- 需要移除、替换、废弃原逻辑/原文案/原配置时，必须优先将原内容注释掉，并在其附近新增调整后的内容。
- 注释原内容时应保持可追溯，避免破坏上下文；除非用户明确要求删除，或原内容会导致编译/运行/校验失败且无法通过注释方式保留。
- 如因语法、平台限制、敏感信息、构建失败等原因必须删除，需先向用户说明原因并确认后再处理。

### 5. 文档编写规范
所有生成的文档末尾必须包含：
- **文档版本**：首次创建 v1.0，小改动递增 v1.1、v1.2，大改动递增 v2.0、v3.0
- **最后更新**：当前日期（格式：YYYY年MM月）
- **整理人**：固定为王新骏

示例格式：
```
---
**文档版本**: v1.0
**最后更新**: 2024年12月
**整理人**: 王新骏
```

## 二、uni-app 开发规范

### 1. 条件编译规范（重要）

**核心原则**：针对不同平台编写不同逻辑时，**必须使用条件编译**，不使用运行时判断。

#### 条件编译指令
- `#ifdef` - 如果定义了某平台
- `#ifndef` - 如果没有定义某平台
- `#endif` - 结束条件编译块

#### 平台标识
- `H5` - H5 平台
- `MP-WEIXIN` - 微信小程序
- `MP-ALIPAY` - 支付宝小程序
- `APP-PLUS` - App

#### 正确示例
```javascript
showBackButton() {
  // #ifdef H5  
  const isOwnerH5 = uni.getStorageSync('isOwnerH5')
  return isOwnerH5 ? false : true
  // #endif
  // #ifndef H5
  return true
  // #endif
}
```

❌ **错误示例**（会产生 Dead Code）：
```javascript
showBackButton() {
  const isOwnerH5 = uni.getStorageSync('isOwnerH5')
  return isOwnerH5 ? false : true
  return true  // 永远不会执行
}
```

#### 在模板中使用
```vue
<!-- #ifdef H5 -->
<view>H5 专属内容</view>
<!-- #endif -->
```

#### 在样式中使用
```scss
/* #ifdef H5 */
.container { padding: 20px; }
/* #endif */
```

#### 条件编译的工作原理

条件编译是**编译时**处理，不是运行时判断：

1. **编译为 H5 时**，编译器会移除 `#ifndef H5` 部分，最终代码：
   ```javascript
   showBackButton() {
     const isOwnerH5 = uni.getStorageSync('isOwnerH5')
     return isOwnerH5 ? false : true
   }
   ```

2. **编译为小程序/APP 时**，编译器会移除 `#ifdef H5` 部分，最终代码：
   ```javascript
   showBackButton() {
     return true
   }
   ```

### 2. 生命周期使用
- **onLoad**：获取页面参数、初始化数据（只触发一次）
- **onShow**：刷新数据（可能从其他页面返回，会多次触发）
- **onReady**：操作 DOM、获取节点信息（只触发一次）
- **mounted**：Vue 组件渲染完成，可操作 refs

### 3. API 使用规范
- `uni.navigateTo()` - 保留当前页面跳转
- `uni.redirectTo()` - 关闭当前页面跳转
- `uni.reLaunch()` - 关闭所有页面跳转
- `uni.switchTab()` - 跳转到 tabBar 页面
- `uni.navigateBack()` - 返回上一页或多级页面

### 4. 样式规范
- 使用 `rpx` 响应式单位（750rpx = 屏幕宽度）
- 避免使用 `*` 选择器
- 小程序不支持本地路径 `background-image`，需用网络图片或 base64
- 不同平台导航栏/状态栏高度不同，需动态获取

#### CSS 属性书写顺序（强制）

按功能分组，组内属性按以下顺序书写，组间空行分隔可选：

```
1. 布局定位
   position / top / right / bottom / left / z-index
   float / clear

2. 盒模型
   display / flex / flex-direction / flex-wrap / justify-content / align-items / align-self
   width / min-width / max-width
   height / min-height / max-height
   margin / margin-top / margin-right / margin-bottom / margin-left
   padding / padding-top / padding-right / padding-bottom / padding-left
   border / border-width / border-style / border-color / border-radius
   box-sizing / overflow

3. 视觉表现
   background / background-color / background-image / background-size / background-position
   color / opacity / visibility / box-shadow

4. 文字排版
   font / font-size / font-weight / font-style / line-height
   text-align / text-decoration / text-overflow / white-space / word-break

5. 动画交互
   transition / transform / animation
   cursor / pointer-events / user-select
```

✅ **正确示例**：
```scss
.card {
  position: relative;
  z-index: 1;

  display: flex;
  align-items: center;
  width: 100%;
  padding: 24rpx;
  border: 1rpx solid #eee;
  border-radius: 8rpx;
  box-sizing: border-box;

  background-color: #fff;
  color: #333;

  font-size: 28rpx;
  line-height: 1.5;

  transition: opacity 0.2s;
}
```

❌ **错误示例**（顺序混乱）：
```scss
.card {
  font-size: 28rpx;
  background-color: #fff;
  position: relative;
  padding: 24rpx;
  color: #333;
  display: flex;
}
```

#### SCSS 嵌套规则（强制）

**核心原则**：使用 `lang="scss"` 时，子元素样式必须嵌套在父元素内，不得全部平铺为独立选择器。

❌ **错误示例**（平铺，禁止）：
```scss
.weigh-bill-tab { padding: 24rpx; }
.weigh-bill-section { margin-bottom: 40rpx; }
.weigh-bill-section-title { font-size: 30rpx; }
.weigh-bill-empty-text { color: #999; }
```

✅ **正确示例**（嵌套，必须）：
```scss
.weigh-bill-tab {
  padding: 24rpx;

  .weigh-bill-section {
    margin-bottom: 40rpx;

    .weigh-bill-section-title { font-size: 30rpx; }

    .weigh-bill-empty {
      .weigh-bill-empty-text { color: #999; }
    }
  }
}
```

嵌套层级与模板 DOM 结构保持一致，便于对照和维护。

### 5. 数据绑定
- `v-for` 必须绑定 `:key`
- 避免模板中使用复杂表达式
- `v-if` 和 `v-show` 在小程序性能差异大

### 6. 页面通信
**事件总线**：
```javascript
// 发送
uni.$emit('update', { data: 'xxx' })
// 监听
uni.$on('update', (data) => {})
// 卸载时移除
onUnload() { uni.$off('update') }
```

**页面栈**：
```javascript
const pages = getCurrentPages()
const prevPage = pages[pages.length - 2]
prevPage.$vm.someMethod()
```

### 7. 性能优化
- 避免在 `onPageScroll` 中执行复杂操作
- 长列表使用虚拟列表或分页
- 图片懒加载
- 控制 setData 频率和数据量

### 8. 跨平台兼容
- H5 可用 `window`、`document`，小程序需条件编译
- 小程序包大小限制：主包 2M，分包总大小 20M

📖 更多详情参考：`docs/uni-app.md`

### 9. 调试技巧

使用条件编译进行平台判断调试：

```javascript
// #ifdef APP-PLUS
console.log('APP')
// #endif

// #ifdef H5
console.log('H5')
// #endif

// #ifdef MP-WEIXIN
console.log('微信小程序')
// #endif
```

## 三、代码调整标准流程

### 流程概览
```
需求理解 → 现状分析 → 影响评估 → 方案设计 → 代码审查 → 实施调整 → 验证测试 → 文档更新
```

### 流程补充：反复交互防遗漏规则（强制）

以下规则用于避免需求梳理和代码调整过程中反复遗漏、误判范围、误用旧字段或误删旧逻辑。

#### 0.1 新信息触发重新评估
当用户补充以下任一信息时，必须回到「需求理解 → 影响评估 → 方案设计」重新检查，不能沿用旧结论直接继续：
- 新增端：PC / 移动端 / 小程序 / H5 / APP
- 新增页面或模块：如查看页、调整页、对账页、付款页
- 新增接口或新字段
- 新增业务范围：如全流程运单、司机对账、运单付款
- 新增实施约束：如旧逻辑只允许注释、不允许删除
- 用户指出“这个也在范围内”或“不是复用旧字段，是新接口新字段”

#### 0.2 需求不得只看标题，必须拆原文
梳理需求时必须逐条拆解需求原文，尤其关注：
- “涉及范围”“其他影响”“测试要求”“其他优化”
- 示例中的边界场景
- 字段初始值、修改后值、保存后值的差异
- 是否包含特殊业务，如全流程运单、分次支付、司机对账

不得只根据标题或当前打开文件判断范围。

#### 0.3 相似场景必须区分触发来源
看似同一个计算逻辑，如果触发来源不同，必须分别确认交互结果。
例如：
- 小计变化导致到付为负
- 回单付变化导致到付为负

即使公式相同，也不能默认处理方式相同；必须确认是否有不同的清空、回退、标红、提示逻辑。

#### 0.4 不确定项必须显式记录
遇到以下情况，不允许脑补实现，必须写入“待确认事项”：
- 后端字段名不确定
- 接口路径不确定
- 新旧字段是否复用不确定
- 小计/合计取值口径不确定
- 保存接口是否需要新增参数不确定
- 页面是否需要前端改动不确定
- 特殊业务数据来源不确定

待确认项需包含：问题、影响范围、当前建议、确认人/接口方（如有）。

#### 0.5 新接口/新字段优先，不得默认复用旧逻辑
如果用户或需求明确“新接口、新字段”，不得默认复用旧字段或旧接口。
旧逻辑只能作为参考，不能作为最终数据来源，除非后端或产品明确确认可复用。

#### 0.6 影响范围必须横向搜索
需求涉及某个展示或字段时，必须搜索同类文案、字段、组件、接口，避免只改当前页面。
检查维度包括：
```
□ 调整页
□ 查看页
□ 列表页
□ 对账页
□ 付款页
□ 移动端对应页面
□ 公共组件
□ 接口封装
□ 历史旧展示
```

#### 0.7 旧逻辑保留规则
修改已有内容时，尤其是隐藏、废弃、替换旧 UI/旧逻辑，原则上不直接删除。
- 模板旧代码：优先注释保留
- JS 旧逻辑：优先注释保留
- 样式旧逻辑：优先注释保留
- 注释旁注明需求号和原因，便于对照

除非旧代码会导致编译失败、运行失败、校验失败，且无法通过注释方式保留，否则不得直接删除。

#### 0.8 改动清单必须包含固定栏目
复杂需求在实施前应输出改动清单，至少包含：
```
□ 需求概述
□ 涉及项目和文件
□ 业务规则
□ 各页面改动点
□ 新增/废弃组件说明
□ 接口和字段说明
□ 不确定/待确认事项
□ 影响范围
□ 验证清单
□ 风险点
□ 开发顺序建议
```

### 1. 需求理解
- 要改什么功能/逻辑
- 为什么要改（业务背景）
- 预期效果

### 2. 现状分析
- 当前实现逻辑
- 涉及的文件和模块
- 业务规则和约束
- 权限控制、数据校验等关键逻辑

⚠️ **注意**：不轻易删除看不懂的代码，先理解业务含义

### 3. 影响范围评估（核心环节）

#### 3.1 直接影响
- 需要修改的文件
- 修改内容和复杂度

#### 3.2 间接影响
检查清单：
```
□ 页面跳转逻辑
□ 数据流转
□ 状态管理
□ 事件监听
□ 路由参数传递
□ 权限控制
□ 条件编译（H5/小程序/APP）
```

#### 3.3 功能影响
- 同模块其他功能
- 跨模块关联功能
- 边界和异常流程
- 用户操作路径

### 4. 技术方案设计

#### 方案选择原则
1. **单一职责**：一个方法只做一件事
2. **逻辑集中**：相关逻辑放在一起
3. **易于维护**：后续修改只需改一处
4. **可扩展性**：预留扩展空间
5. **最小改动**：改动越小越好

#### 示例对比

❌ **不推荐**（逻辑分散）：
```javascript
@tap.stop="isLowRisk(item.riskLevel) ? null : gotoDetails(item)"
```

✅ **推荐**（逻辑集中）：
```javascript
// 模板简洁
@tap.stop="gotoDetails(item)"

// 方法内部处理
gotoDetails(item) {
  if (this.isLowRisk(item.riskLevel)) return
  // 原有逻辑
}
```

### 5. 代码审查

审查要点：
```
□ 权限控制逻辑（不能删除）
□ 数据校验逻辑（不能删除）
□ 状态判断逻辑（需要保留）
□ 条件编译（需要保留）
□ 业务规则约束（需要保留）
```

### 6. 验证测试

#### 6.1 功能验证清单
```
□ 修改功能正常
□ 原有功能不受影响
□ 边界情况正常
□ 异常提示合理
□ 上下游调用正常
□ 数据流转正常
□ 平台兼容性（H5/小程序/APP）
```

#### 6.2 验证方法
1. 手动测试：逐一验证测试用例
2. 交叉验证：不同状态、不同平台
3. 回归测试：验证原功能
4. 边界测试：极端和异常情况

### 7. 关键原则
1. **安全第一**：不确定不做
2. **影响最小**：能不改不改
3. **逻辑集中**：相关逻辑放一起
4. **充分验证**：改完必测
5. **及时回滚**：发现问题立即回滚

### 8. 常见错误
| 错误 | 后果 | 正确做法 |
|-----|------|---------|
| 直接修改，不评估影响 | 改一处坏一片 | 先评估影响范围 |
| 删除看不懂的代码 | 破坏关键逻辑 | 先理解代码作用 |
| 模板写复杂逻辑 | 难以维护 | 逻辑放方法中 |
| 改完不测试 | 问题延迟到生产 | 改完立即验证 |
| 一次改多个功能 | 问题难定位 | 一次只改一个 |

### 9. 提交前检查清单
```
□ 需求理解清晰
□ 现状分析完整
□ 影响范围已评估
□ 技术方案已确定
□ 代码审查已完成
□ 功能已验证
□ 平台兼容性已验证
□ 文档已更新
□ 无遗留问题
```

### 10. 实战案例

**案例：低风险时风险提示条仅展示不支持跳转**

1. **需求理解**：低风险运单的风险提示条只展示，不支持点击跳转
2. **现状分析**：当前所有风险等级都可以点击跳转
3. **影响范围评估**：
   - 直接影响：`orderList.vue` 的风险提示条
   - 间接影响：风险详情页（不受影响）
   - 功能影响：多选提交、其他跳转功能（不受影响）
4. **技术方案**：在 `gotoRiskDetails` 方法内部判断，低风险直接返回
5. **代码审查**：确认 `isView` 判断逻辑需要保留
6. **实施调整**：
   - 模板：低风险不显示右箭头
   - 方法：新增 `isLowRisk()` 判断
   - 方法：修改 `gotoRiskDetails()` 拦截低风险
7. **验证测试**：
   - 低风险：不显示箭头，点击无反应 ✓
   - 中高风险：正常跳转 ✓
   - 多选提交：正常工作 ✓
   - 其他功能：不受影响 ✓
8. **文档更新**：更新操作说明

**总结**：通过完整的流程，确保了改动的安全性和可控性。

## 四、代码质量验证

### 1. ESLint 验证要求（强制）

每次代码修改后必须运行 eslint 验证，确保通过（无 error）。

#### 验证命令
```bash
# 单文件
cd logisticsweb
npx eslint src/path/to/file.vue --no-eslintrc -c .eslintrc.js

# 自动修复
npx eslint src/path/to/file.vue --no-eslintrc -c .eslintrc.js --fix
```

#### 验证时机
- ✅ 代码修改完成后
- ✅ 重构代码后
- ✅ 提交代码前

#### 处理原则
1. 优先使用 `--fix` 自动修复格式问题
2. 手动修复逻辑问题（未使用变量、缺少 key 等）
3. 只修复本次修改引入的问题，不修复历史遗留
4. 所有 error 必须清零

#### 常见问题速查

| 问题 | 修复方式 |
|-----|---------|
| `no-trailing-spaces` | `--fix` 自动修复 |
| `indent` | `--fix` 自动修复 |
| `no-unused-vars` | 手动删除变量 |
| `vue/require-v-for-key` | 手动添加 `:key` |

### 2. 验证清单
```
□ 已运行 eslint 验证
□ 所有 error 已修复
□ 验证通过（exit code = 0）
□ 代码格式符合规范
```

### 3. 特殊情况处理

**历史遗留问题**：
- 如果文件原本就有 eslint 错误，只修复本次修改引入的问题
- 可以通过对比修改前后的 eslint 输出来判断

**无法自动修复的问题**：
- 优先手动修复
- 如果确实需要保留，添加 `eslint-disable-next-line` 注释并说明原因
- 与团队确认是否可以调整 eslint 规则

**批量修改场景**：
- 一次性验证所有修改的文件
- 使用 `--fix` 批量自动修复
- 手动处理剩余问题后再次批量验证

**紧急情况**：
- 即使是紧急修复，也必须通过 eslint 验证
- 不允许因为紧急而跳过代码质量检查

## 五、执行过程可见性规则（强制）

> 目的：让用户能在 AI 动手前/动手后直接看到执行状态，避免漏改、误改、反复纠正。

### 1. 接单确认（收到需求后，动手前必须输出）

收到任何涉及代码修改的需求后，**必须先输出以下结构，再开始实施**：

```
【接单确认】
需求理解：xxx
涉及文件：
  - xxx（角色：调整页 / 查看页 / 列表页 / 移动端 / 公共组件 / 接口封装）
横向搜索已检查：
  □ 调整页    □ 查看页    □ 列表页
  □ 对账页    □ 付款页    □ 移动端对应页面
  □ 公共组件  □ 接口封装  □ 历史旧展示
旧逻辑处理方式：注释保留 / 不涉及
不确定项（需确认后再动手）：
  - 问题 1：xxx → 影响：xxx
  - 问题 2：xxx → 影响：xxx
是否可以开始实施？（有不确定项时等用户确认）
```

- 有不确定项时，**必须等用户确认后再动手**，不得自行猜测实现。
- 横向搜索必须实际执行（用搜索工具），不得凭印象填写。

### 2. 完成自检（每次修改完成后必须输出）

代码修改完成后，**必须输出以下自检结果**：

```
【完成自检】
□ 横向搜索已完成，涉及页面全部处理：是 / 否（遗漏：xxx）
□ 旧逻辑已注释保留，未直接删除：是 / 否（例外原因：xxx）
□ 不确定字段/接口已列出，未自行猜测：是 / 否
□ ESLint 验证已通过（exit code = 0）：是 / 否
□ 新信息出现时已重新评估影响范围：是 / 否 / 不适用
遗留问题：xxx（无则填"无"）
```

### 3. 触发重新评估的强制动作

用户补充以下任一信息时，**必须在回复开头声明"重新评估范围"，并重新输出接单确认**，不得沿用旧结论：

- 新增端（PC / 移动端 / 小程序 / H5 / APP）
- 新增页面或模块
- 新增接口或新字段
- 指出"这个也在范围内"
- 指出"不是复用旧字段，是新接口新字段"

### 4. AI 历史高频失误提醒

以下是过去反复发生、需要每次主动防范的错误：

| 失误类型 | 具体表现 | 防范动作 |
|---------|---------|---------|
| 范围遗漏 | 只改当前文件，遗漏查看页/移动端/公共组件 | 接单时强制执行横向搜索 |
| 直接删除旧代码 | 替换逻辑时删掉原代码，无法追溯 | 改前默认注释保留，不删除 |
| 沿用旧结论 | 用户补充新信息后继续按旧方案执行 | 新信息 → 强制重新评估 |
| 复用旧字段 | 未确认直接套用旧接口/旧字段 | 优先列为不确定项，等用户确认 |
| 脑补不确定项 | 字段名/接口路径不确定时自行猜测 | 不确定 → 写入待确认，不动手 |
| 来源不同但默认行为相同 | 相同公式不同触发来源，默认处理方式一样 | 分别列出，逐一确认 |
| 只看需求标题 | 遗漏"其他影响""测试要求"等附加内容 | 必须逐条拆解需求原文 |
