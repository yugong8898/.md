# AI 多角色协作工作流

## 文件清单

### 角色文件

| 角色 | 文件 | 职责 | 输出 |
|-----|------|------|------|
| 全链路编排器 | `role-workflow.md` | 串联全链路流程，断点续传 | 各阶段产物 + 进度文件 |
| 需求分析师 | `role-pm.md` | 拆解需求，识别不确定项 | 需求分析文档 |
| 架构师 | `role-architect.md` | 交互式设计技术方案，横向排查影响范围 | 技术方案文档 |
| 开发工程师 | `role-coder.md` | 按方案写代码，旧逻辑注释保留 | 代码 + 完成自检 |
| 代码审查员 | `role-reviewer.md` | 对比需求和代码，找问题 | 审查报告（P0/P1/P2分级） |
| 重构师 | `role-refactor.md` | 不改行为，提升代码可读性和结构 | 重构前分析 + 代码 + 自检 |
| Bug 侦探 | `role-bugdetective.md` | 定位 bug 根因，再给修复方案 | 排查报告 + 根因确认 + 修复 |
| 提测清单生成器 | `role-qa-checklist.md` | 根据改动生成结构化提测清单 | 提测清单文档 |
| UI 还原审查员 | `role-ui-reviewer.md` | 对比设计稿和实际页面截图，找还原偏差 | UI审查报告（P0/P1/P2分级） |

> 每个角色文件内置了所有项目的技术栈对应关系，AI 根据文件路径自动识别，**无需额外引用 stack 文件**。需要手动查阅时，见 [`stack-index.md`](stack-index.md)。

## 全自动流程（推荐）

### 一键启动全链路

```
@role-workflow.md 走流程，需求是 xxx
```

编排器自动串联以下 6 个阶段，每阶段产物保存到 `.md/temp/`，支持断点续传：

```
阶段0 需求分析 → 阶段1 代码分析 → 阶段2 技术方案 → 阶段3 编码 → 阶段4 审查 → 阶段5 提测清单
```

**编排器特性**：
- 自动检测当前进度，从断点继续
- 每阶段产物自动保存到 `.md/temp/`
- 进度面板可视化（回复"当前进度"查看）
- 支持跳过/重做指定阶段
- 有待确认项时强制暂停，不接受裸"继续"
- 审查发现 P0 暂停等用户修复，不自动改代码

**编排器指令速查**：

| 指令 | 作用 |
|------|------|
| `@role-workflow.md 走流程，需求是 xxx` | 全新启动 |
| `@role-workflow.md 继续` | 从断点继续 |
| `@role-workflow.md 当前进度` | 查看进度面板 |
| `@role-workflow.md 从阶段2开始` | 指定阶段开始 |
| `@role-workflow.md 重做阶段4` | 重新执行指定阶段 |
| `@role-workflow.md 归档` | 产物归档到 `.md/需求/` |
| `@role-workflow.md 清理` | 清理临时文件 |

## 手动模式（按需逐步触发）

### 第一步：需求分析

1. 引用 `@role-pm.md`，粘贴需求原文
2. 等 AI 输出需求分析文档
3. **有不确定项时，先和产品/后端确认，再进行下一步**
4. 把输出内容保存到 `.md/temp/需求文档-xxx.md`

### 第二步：技术方案

1. 引用 `@role-architect.md` + `@需求文档-xxx.md`
2. 等 AI 输出技术方案（AI 自动识别技术栈）
3. **有不确定项时，先确认，再进行下一步**
4. 把输出内容保存到 `.md/temp/技术方案-xxx.md`

### 第三步：写代码

1. 引用 `@role-coder.md` + `@技术方案-xxx.md`
2. 告诉 AI：`请先处理文件 [文件路径]，改动点：xxx`
3. 等 AI 写完，**检查完成自检里的每一项**
4. 对下一个文件重复这个步骤

### 第四步：代码审查

1. 引用 `@role-reviewer.md` + `@需求文档-xxx.md` + `@技术方案-xxx.md` + `@改动的文件`
2. 等 AI 输出审查报告
3. 按报告修复 P0 和 P1 问题

### 第五步：生成提测清单

1. 引用 `@role-qa-checklist.md` + `@需求文档-xxx.md` + `@技术方案-xxx.md`
2. 等 AI 输出提测清单
3. 检查用例是否覆盖所有改动点，确认无误后发给 QA

## 其他流程

### 重构

1. 引用 `@role-refactor.md` + `@要重构的文件路径`
2. 等 AI 输出重构前分析报告
3. 确认方案后回复"确认，开始重构"
4. 每次只做一个重构维度，完成后检查自检

### Bug 排查

1. 引用 `@role-bugdetective.md`，描述 bug 现象（含复现步骤、平台、报错信息）
2. 配合 AI 追问，提供验证结果，逐步缩小范围
3. 根因确认后再让 AI 给修复代码

### UI 还原审查

1. 准备设计稿截图 + 实际页面截图
2. 引用 `@role-ui-reviewer.md`，提供截图
3. 等 AI 输出 UI 还原审查报告（P0/P1/P2 分级）

### 代码理解（接手老模块时）

1. 引用 `@code-analyzer`，指定要分析的文件路径
2. AI 输出模块分析报告（数据流、事件流、组件交互、依赖关系、注意事项）

## 中转文件目录

```
.md/temp/
├── workflow-state.md          ← 流程进度文件（编排器自动维护）
├── 需求文档-[需求名].md        ← 阶段0输出
├── 代码分析-[需求名].md        ← 阶段1输出
├── 技术方案-[需求名].md        ← 阶段2输出
├── 审查报告-[需求名].md        ← 阶段4输出
└── 提测清单-[需求名].md        ← 阶段5输出
```

做完后可以删掉，或者归档到 `.md/需求/` 目录。

## 快速参考：什么情况用哪个

| 任务类型 | 推荐方式 |
|---------|---------|
| 复杂新需求（多页面/多端） | `@role-workflow.md` 全自动 |
| 中等改动（1-3个文件） | `@role-workflow.md` 全自动 |
| 小改动（1个文件，逻辑简单） | `@role-coder.md` → `@role-qa-checklist.md` |
| 纯样式调整 | `@role-coder.md` |
| 整理/优化旧代码 | `@role-refactor.md` |
| 出现 bug，原因不明 | `@role-bugdetective.md` |
| 出现 bug，原因清楚 | `@role-coder.md`（直接修） |
| 开发完成，准备提测 | `@role-qa-checklist.md` |
| 接手陌生模块，需要读懂代码 | `@code-analyzer`（模块分析模式） |
| 设计稿还原度审查 | `@role-ui-reviewer.md` |
| 查看当前流程进度 | `@role-workflow.md 当前进度` |

## 注意事项

- 全自动模式下，每个阶段完成后仍需用户确认"继续"才推进
- 每次对话只激活一个角色，不要在同一个 Chat 里混用多个角色
- `role-coder.md` 每次只处理一个文件，处理完再处理下一个
- 看到 `TODO:` 标记就是有未确认项，需要手动确认后再继续
- 审查报告里 P0 是阻塞项，必须修复后才能提交
- 产物文件统一保存到 `.md/temp/`，完成后可归档或清理
