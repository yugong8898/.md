# AI 多角色协作工作流

## 文件清单

### 七个角色文件

| 角色 | 文件 | 职责 | 输出 |
|-----|------|------|------|
| 需求分析师 | `role-pm.md` | 拆解需求，识别不确定项 | 需求分析文档 |
| 架构师 | `role-architect.md` | 设计技术方案，横向排查影响范围 | 技术方案文档 |
| 开发工程师 | `role-coder.md` | 按方案写代码，旧逻辑注释保留 | 代码 + 完成自检 |
| 代码审查员 | `role-reviewer.md` | 对比需求和代码，找问题 | 审查报告 |
| 重构师 | `role-refactor.md` | 不改行为，提升代码可读性和结构 | 重构前分析 + 代码 + 自检 |
| Bug 侦探 | `role-bugdetective.md` | 定位 bug 根因，再给修复方案 | 排查报告 + 根因确认 + 修复 |
| 提测清单生成器 | `role-qa-checklist.md` | 根据改动生成结构化提测清单 | 提测清单文档 |

> 每个角色文件内置了所有项目的技术栈对应关系，AI 根据文件路径自动识别，**无需额外引用 stack 文件**。

## 使用步骤（照着做）

### 第一步：需求分析

1. Cursor 里打开一个新的 Chat（`Cmd+L`）
2. 引用 `@role-pm.md`，粘贴需求原文
3. 等 AI 输出需求分析文档
4. **有不确定项时，先和产品/后端确认，再进行下一步**
5. 把输出内容保存到 `.md/temp/需求文档-xxx.md`

### 第二步：技术方案

1. 新开一个 Chat
2. 引用 `@role-architect.md` + `@需求文档-xxx.md`
3. 等 AI 输出技术方案（AI 自动识别技术栈）
4. **有不确定项时，先确认，再进行下一步**
5. 把输出内容保存到 `.md/temp/技术方案-xxx.md`

### 第三步：写代码

1. 新开一个 Chat（或用 Composer `Cmd+I`）
2. 引用 `@role-coder.md` + `@技术方案-xxx.md`
3. 告诉 AI：`请先处理文件 [文件路径]，改动点：xxx`
4. 等 AI 写完，**检查完成自检里的每一项**
5. 对下一个文件重复这个步骤

### 第四步：代码审查

1. 新开一个 Chat
2. 引用 `@role-reviewer.md` + `@需求文档-xxx.md` + `@技术方案-xxx.md` + `@改动的文件`
3. 等 AI 输出审查报告
4. 按报告修复 P0 和 P1 问题

### 第五步：生成提测清单

1. 新开一个 Chat
2. 引用 `@role-qa-checklist.md` + `@需求文档-xxx.md` + `@技术方案-xxx.md`
3. 等 AI 输出提测清单
4. 检查用例是否覆盖所有改动点，确认无误后发给 QA

## 重构流程

1. 新开一个 Chat
2. 引用 `@role-refactor.md` + `@要重构的文件路径`
3. 描述重构目标，等 AI 输出**重构前分析报告**
4. 确认方案后回复"确认，开始重构"
5. 每次只做一个重构维度，完成后检查自检

---

## Bug 排查流程

1. 新开一个 Chat
2. 引用 `@role-bugdetective.md`，描述 bug 现象（含复现步骤、平台、报错信息）
3. 配合 AI 追问，提供验证结果，逐步缩小范围
4. 根因确认后再让 AI 给修复代码

---

## 中转文件目录

每次做新需求，在 `.md/temp/` 下建两个文件：

```
.md/temp/
├── 需求文档-[需求名].md    ← 角色1的输出
└── 技术方案-[需求名].md   ← 角色2的输出
```

做完后可以删掉，或者归档到 `.md/需求/` 目录。

## 快速参考：什么情况用哪个角色

| 任务类型 | 用哪个角色 |
|---------|----------|
| 复杂新需求（多页面） | PM → 架构师 → 开发 → 审查 → 提测清单（全流程） |
| 中等改动（1-3个文件） | 架构师 → 开发 → 审查 → 提测清单 |
| 小改动（1个文件，逻辑简单） | 开发 → 提测清单 |
| 纯样式调整 | 开发 |
| 整理/优化旧代码 | 重构师 |
| 出现 bug，原因不明 | Bug 侦探 |
| 出现 bug，原因清楚 | 开发（直接修） |
| 开发完成，准备提测 | 提测清单生成器 |

---

## 注意事项

- 每次对话只激活一个角色，不要在同一个 Chat 里混用多个角色
- 角色3每次只处理一个文件，处理完再处理下一个
- 看到 `TODO:` 标记就是有未确认项，需要手动确认后再继续
- 审查报告里 P0 是阻塞项，必须修复后才能提交
