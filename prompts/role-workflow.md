# 全链路研发流程编排器（Workflow）

## 激活方式

直接 `@role-workflow.md`，描述需求或使用以下指令：

```
@role-workflow.md 走流程，需求是 xxx
@role-workflow.md 帮我跑完整流程
@role-workflow.md 继续（从断点继续）
@role-workflow.md 当前进度
@role-workflow.md 从阶段3开始
```

---

## 与 wlyd-workflow Skill 的关系

本文件是 `wlyd-workflow` Skill 的 Prompt 版等价物，用于手动多 Chat 场景（`@role-workflow.md`）。两者编排逻辑一致，**以 `wlyd-workflow` Skill 为准**。修改任一处的阶段编排逻辑时，需同步另一处，避免双编排器不一致。

---

## 核心理念

**工具不该是断开的。** 把分散的需求分析、代码分析、技术方案、编码、审查、提测等环节自动串联，让用户只需说"走流程"，就能从需求到提测一路推进，每个阶段自动衔接、自动保存、支持断点续传。

---

## 全流程阶段

```
阶段0：需求分析       → role-pm / wlyd-requirement-change-analysis
  ↓
阶段1：代码分析       → code-analyzer（模块分析模式）
  ↓
阶段2：技术方案       → role-architect
  ↓
阶段3：编码执行       → role-coder（逐文件）
  ↓
阶段4：代码审查       → role-reviewer / wlyd-review
  ↓
阶段5：提测清单       → role-qa-checklist
```

每个阶段的**输出产物**保存到 `.md/temp/` 目录，作为下一阶段的输入，也作为断点续传的标记。

---

## 角色文件加载机制

各阶段的执行逻辑定义在角色文件（`.md/prompts/`）或 Skill 文件中。编排器本身不包含各角色的具体逻辑，而是负责**调度和衔接**。

进入某个阶段时，读取对应的 `.md/prompts/role-xxx.md` 文件内容，按文件中定义的角色定义、输出格式、强制规则执行。

### Qoder 与 Cursor 的差异

| 维度 | Qoder | Cursor |
|------|-------|--------|
| 编排方式 | 单对话全自动，Agent 读取角色文件并按阶段执行 | 多 Chat 手动，每个阶段新开 Chat `@role-xxx.md` |
| 状态衔接 | 编排器自动读写 workflow-state.md | 人工查看进度文件后手动衔接 |
| 适用场景 | 中小需求全自动跑完 | 复杂需求逐阶段深度交互 |

## 执行逻辑

### 第一步：检测当前进度

检查 `.md/temp/workflow-state.md` 是否存在。存在则读取并输出进度面板，询问是否从断点继续；不存在则从阶段 0 开始。

### 第二步：按阶段推进

进入每个阶段前，先检查前置条件是否满足（上一阶段的产物是否存在）。如果前置产物不存在，提示用户先完成上一阶段，或手动补充输入。

---

## 各阶段执行规则

### 阶段 0：需求分析

**前置条件**：用户提供需求原文（文字、文件路径、截图说明）

**执行**：
1. 读取需求原文
2. 判断需求复杂度：
   - 简单需求（1-2个文件）→ 使用 `role-pm` 输出精简需求分析
   - 复杂需求（多文件/多端/多模块）→ 使用 `wlyd-requirement-change-analysis` 输出完整改动清单
3. 输出需求分析文档

**产物**：`.md/temp/需求文档-[需求名].md`

**完成后**：
1. 保存产物，更新 `workflow-state.md`：标记阶段 0 完成
2. 检查不确定项追踪表：若存在状态为"待确认"的项，**必须输出警告并暂停**：
   ```
   ⚠️ 发现 N 个待确认项，必须先确认后再继续：
   - 问题1：xxx → 影响：xxx
   - 问题2：xxx → 影响：xxx
   请确认后回复"已确认，继续"推进到下一阶段。
   回复裸"继续"将被忽略。
   ```
3. 无待确认项时输出进度面板，提示用户回复"继续"进入代码分析

### 阶段 1：代码分析

**前置条件**：阶段 0 产物存在

**执行**：
1. 从需求文档中提取涉及的文件路径
2. 对每个关键文件使用 `code-analyzer` 模块分析模式
3. 输出现状分析报告

**产物**：`.md/temp/代码分析-[需求名].md`

**完成后**：保存产物，更新进度，输出进度面板，提示回复"继续"进入技术方案

### 阶段 2：技术方案

**前置条件**：阶段 0 + 阶段 1 产物存在

**执行**：
1. 引用需求文档 + 代码分析报告作为输入
2. 使用 `role-architect` 进行技术方案设计
   - 简单需求 → 快速模式
   - 复杂需求 → 完整模式
3. 输出技术方案文档

**产物**：`.md/temp/技术方案-[需求名].md`

**完成后**：
1. 保存产物，更新 `workflow-state.md`：标记阶段 2 完成
2. 检查不确定项追踪表：若存在状态为"待确认"的项，**必须输出警告并暂停**：
   ```
   ⚠️ 发现 N 个待确认项，必须先确认后再进入编码：
   - 问题1：xxx → 影响文件：xxx
   - 问题2：xxx → 影响文件：xxx
   不确定项直接影响编码实现，请确认后回复"已确认，继续"。
   回复裸"继续"将被忽略。
   ```
3. 无待确认项时输出进度面板，提示用户回复"继续"进入编码

### 阶段 3：编码执行

**前置条件**：阶段 2 产物存在

**执行**：
1. 从技术方案"开发顺序建议"章节提取文件清单和执行顺序
2. 使用 `role-coder` 逐文件执行
3. 每个文件改完后输出完成自检，等用户确认后再处理下一个

**产物**：代码变更（git diff）

**完成后**：更新进度，记录已处理文件列表，输出进度面板，提示回复"继续"进入代码审查

### 阶段 4：代码审查

**前置条件**：阶段 2 + 阶段 3 产物存在

**执行**：
1. 使用 `role-reviewer` 对比需求文档 + 技术方案 + 代码改动
2. 输出审查报告（P0/P1/P2 分级）
3. 如果有 P0 问题，**暂停并输出修复提示**：
   ```
   🔴 发现 N 个 P0 问题，必须修复后才能继续：
   - [文件路径] 问题描述 → 修复建议
   请修复后回复"已修复，重新审查"，编排器将重新执行阶段4。
   ```
   不自动修改代码，等用户确认修复完成后重跑审查。

**产物**：`.md/temp/审查报告-[需求名].md`

**完成后**：保存产物，更新进度，输出进度面板，提示回复"继续"生成提测清单

### 阶段 5：提测清单

**前置条件**：阶段 0 + 阶段 4 产物存在

**执行**：
1. 使用 `role-qa-checklist` 生成提测清单
2. 汇总改动范围、主流程用例、边界用例、回归用例

**产物**：`.md/temp/提测清单-[需求名].md`

**完成后**：保存产物，更新进度为全部完成，输出最终进度面板，提示运行 ESLint 验证后提交代码

## 特殊指令

| 指令 | 作用 |
|------|------|
| `继续` | 从当前阶段推进到下一阶段 |
| `当前进度` | 输出进度面板，不执行任何操作 |
| `重做阶段N` | 重新执行指定阶段（保留其他阶段产物） |
| `跳过阶段N` | 跳过指定阶段（需说明原因） |
| `从阶段N开始` | 直接从指定阶段开始 |
| `归档` | 将 `.md/temp/` 下的产物归档到 `.md/需求/` 目录 |
| `清理` | 删除 `.md/temp/` 下的临时文件和进度文件 |

---

## 断点续传规则

1. 每个阶段完成后**必须**更新 `workflow-state.md`
2. 用户中断后重新激活时，自动读取进度文件
3. 如果进度文件损坏或不存在，询问用户从哪个阶段开始
4. 如果某阶段产物文件丢失，提示用户重新执行该阶段

---

## 与各工具的衔接关系

| 阶段 | 调用工具 | 输入 | 输出产物 |
|------|---------|------|---------|
| 0 | role-pm / wlyd-requirement-change-analysis | 需求原文 | 需求文档 |
| 1 | code-analyzer（模块分析模式） | 需求文档中的文件路径 | 代码分析报告 |
| 2 | role-architect | 需求文档 + 代码分析报告 | 技术方案 |
| 3 | role-coder | 技术方案 | 代码变更 |
| 4 | role-reviewer | 需求文档 + 技术方案 + 代码变更 | 审查报告 |
| 5 | role-qa-checklist | 需求文档 + 技术方案 + 审查报告 | 提测清单 |

---

## 注意事项

- 每个阶段完成后必须等用户确认"继续"再推进，不自动跳过
- 阶段0/阶段2完成后，有待确认项时强制暂停，不接受裸"继续"，必须回复"已确认，继续"
- 编码阶段（阶段3）必须逐文件执行，执行顺序按技术方案"开发顺序建议"章节，每文件完成后自检
- 审查阶段发现 P0 问题时，暂停并列出问题清单，等用户修复后回复"已修复，重新审查"，再重跑阶段4
- 全流程完成后，提示用户运行 ESLint 验证
- 产物文件统一保存到 `.md/temp/` 目录，完成后可归档或清理
