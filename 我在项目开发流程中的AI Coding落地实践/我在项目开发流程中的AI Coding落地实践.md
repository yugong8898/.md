# 我在项目开发流程中的AI Coding落地实践

> 分享人：刘冰奥



前面几次分享，同事们聊了大量AI Coding实操踩坑，有“激进型”、也有”保守型“的、有觉得简单需求可以，复杂需求就”管不住“等。但复盘下来发现：我们之前都聚焦在编码写代码这单一环节，忽略了**编码前、编码中、编码后全链路的原生研发痛点**。很多AI用不好，不是AI编码能力差，而是我们没用AI解决研发上游的低效问题。



今天和大家探讨两个问题：

①**我们日常开发中，到底有哪些痛点？**

②**每个痛点AI能帮我们做什么？**



---

## 研发链路几个痛点

### 痛点1：存量业务与代码陌生，代码溯源定位低效

接手老需求、迭代历史页面、接手他人模块时，不清楚业务流转逻辑、不清楚代码分层、不知道页面关联接口；单页面溯源、读懂历史冗余代码平均耗时太长，跨模块溯源难度翻倍

### 痛点2：需求落地低效：PRD拆解、疑点梳理、任务拆分

产品PRD表述模糊、交互分散在原型备注；开发需要反复核对产品、自行拆分前后端任务、梳理异常交互场景；经常出现需求理解偏差，导致后期返工

### 痛点3：技术设计耗时、设计文档不规范，甚至没有技术设计

- 简单需求不用设计

- 复杂需求设计太耗时

- 没有设计直接上手写代码

### 痛点4：AI Coding编码环节固有卡点\-\-代码质量

看似AI跑完显示代码实现完成，实际边界逻辑、异常捕获、兼容逻辑缺失；代码改动范围不可控，隐性修改无关文件；幻觉导致越改bug越多；心理层面不敢大规模合并，必须逐行review

### 痛点5：AI产出代码风格混乱，无统一开发规范

- 每个人提示词、AI模型不同，同类型功能出现多种编码写法：请求封装、错误处理、表单校验、注释规范不统一；代码评审耗时暴涨，后续维护成本升高

- **AI写了啥，我也不知道**

## 我的AI工程师团队\-\-Agent\+Skill\+规则/记忆系统





### **1\. 代码分析（code\-analyzer）**\-\-自定义Agent

[code\-analyzer\.md](图片和附件/code-analyzer.md)

接到需求，第一步不是写代码，而是让AI帮我读懂现有代码。

比如需求说"在司机管理列表加一个上传按钮"，我不会直接让AI开写。我会先问：运力列表和司机管理现在的实现逻辑是什么？有哪些字段、权限怎么控制的、数据流怎么走？

**目的只有一个：让AI有上下文，而不是盲人摸象。**

### **2\. 需求拆分（fe\-req\-splitter）**\-\-自定义Agent

[fe\-req\-splitter\.md](图片和附件/fe-req-splitter.md)

读懂代码之后，把需求拆成可执行的任务清单，带优先级、带风险点、带工作量评估。

还是刚才那个上传按钮，拆出来可能是：按钮UI新增、权限配置对接、上传接口联调、错误处理、文件大小限制、格式校验。拆清楚之后，人和AI都知道下一步要做什么。

### **3\. 技术方案设计（tech\-design）**\-\-自定义skill

[SKILL\.md](图片和附件/SKILL.md)

**这一步最关键——人必须深度参与，有些决策不能做、也不能交给AI。**

组件怎么拆、接口怎么定、状态怎么管、异常怎么处理，这些架构层面的决策必须人来做。AI的作用是：你把思路说清楚，它帮你输出成结构化的方案文档，自动存到固定目录。

举个例子，一个表单提交的实现，人要先定好逻辑骨架：

```JavaScript
handleSubmit() {
  // 1. 校验表单
  // 2. 整理参数
  // 3. 调用提交接口
  // 4. 保存成功提示用户，跳转列表
  // 5. 异常处理
}
```

AI负责按照这个骨架，把详细的方案文档写规范。**人定架构，AI补细节。**

### **4\. 编码执行（Plan \+ 规则系统）**\-\-系统Agent

AI开始写代码之前，必须先有Plan——也就是执行计划。更重要的是，得有一套**规则系统**约束它：方法注释怎么写、模块怎么拆分、变量怎么命名、错误怎么处理。

没有规则的AI，就像一个没有代码规范的新手程序员——写得快，但烂得快。
说到底就是一句话：**用标准化的流程，把AI的不确定性，变成确定性效率。**

### **5\. Git提交（Git\-commit）**\-\-自定义Skill

[SKILL\.md](图片和附件/SKILL%202.md)

代码写完后，AI自动分析变更内容，按Angular规范生成commit信息，带上任务ID。这件事很机械，但团队里经常有人懒得写或者写得很乱，交给AI很合适。

### **6\. Code Review \& 提测（****branch\-diff\-reviewer/****Code\-review）**\-\-自定义Agent

[branch\-diff\-reviewer\.md](图片和附件/branch-diff-reviewer.md)

最后一步，AI帮我整理本次改动的主流程代码，生成Review报告；同时自动生成提测单，列清楚改了哪些地方、重点测什么。

**注意：AI的Review不能替代人的Review。** 它帮你整理材料、做初筛，最终把关还得是人。

**所以整个流程跑下来，人只做三件事：定方案、做关键决策、最后检查。剩下的交给AI。**

### 流程比工具更重要

前面说的那些工具，单独看都不复杂。**真正的难点在于：怎么让它们串成一个稳定、可复用的流程。**

复杂需求涉及多个模块联动，业务逻辑比较复杂，不是AI直接能搞定的。

我的完整流程是这样的：

1. **先分析**：用code\-analyzer把相关模块的代码逻辑理清楚；

2. **再拆分**：用fe\-req\-splitter把需求拆成十几项具体任务，标出依赖关系和技术风险；

3. **人定方案**：我定架构，tech\-design输出结构化的技术方案文档；

4. **AI执行**：按Plan推进编码，全程受规则系统约束，确保代码风格、模块拆分、注释规范都统一；

5. **收尾自动化**：Git\-commit自动提交，Code\-review输出报告，最后生成提测单。

整个过程中，**人只做三件事：定方案、做关键决策、最后检查。剩下的交给AI。**



### 总结

依托 Agent\+Skill 能力，搭建从代码分析、需求拆分、技术设计到编码、提测的 AI 全链路研发流程；人把控架构决策与最终核验，AI 承接全链路重复性工作，通过标准化规则消解 AI 不确定性，实现提效。
