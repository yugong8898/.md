### 2. P2-1 函数命名冲突修正 ✅

**修改位置**：第七章 P2-1 的 `costCalculation.js`

**问题**：模块级函数 `count` 与 mixin 方法 `count` 同名，存在递归调用风险

**修订后**：
- 模块级函数改名为 `calcFreightTotal`
- mixin 方法改为调用 `calcFreightTotal(Amount, Rate, this.roundingMode)`

```js
// 修订前
export function count(amount, rate, roundingMode) { ... }
methods: {
  count(Amount, Rate) {
    return count(Amount, Rate, this.roundingMode)  // ← 潜在递归风险
  }
}

// 修订后
export function calcFreightTotal(amount, rate, roundingMode) { ... }
methods: {
  count(Amount, Rate) {
    return calcFreightTotal(Amount, Rate, this.roundingMode)  // ← 明确调用模块函数
  }
}
```

### 3. P0-2 补充 beforeDestroy 清理逻辑 ✅

**修改位置**：第五章 P0-2 debounce 写法修复

**新增内容**：
```js
// created 中创建
created() {
  this._debouncedDealZero = debounce(() => {
    this.dealZeroModeFn()
  }, 500)
}

// 新增：组件销毁时清理
beforeDestroy() {
  this._debouncedDealZero && this._debouncedDealZero.cancel()
}
```

**补充说明**：
- `_debouncedDealZero` 不需要在 data() 中声明（实例属性，非响应式）
- 项目使用的 debounce 假设为 lodash debounce（支持 .cancel()）
- 如果是自实现 debounce，需调整清理方式

### 8. P0-5 补充行为变更说明 ✅

**修改位置**：第五章 P0-5 commit() 修复

**新增"行为变更说明"章节**：
```
⚠️ 行为变更说明：
原 forEach 实现中，发现重复编码后 `n = '1'` + `return false`（无效），forEach 继续执行，
后续行若存在"费用类型未填写"会覆盖 `n = '2'`，最终可能报"未填写"而非"重复"。

改为 `break outer` 后，发现重复编码立即中断，`n` 不会被后续行覆盖，始终报"重复编码"。

这是预期的行为优化（重复编码比未填写更具体），符合业务逻辑，且原实现的覆盖行为本身是 bug。
```
