# 业务归属公司数据来源对比分析

## 一、核心结论

### 是否同一数据源？

**答案：❌ 不是同一数据源**

| 对比维度 | DeptHierarchyTreeSelect 组件 | 用户管理详情页面 |
|---------|----------------------------|----------------|
| **数据接口** | `/PlatformUmUserbaseinfo/getLecUserCompanyRelByUserBaseId` | `/contract-company-page/getContractOnlineDropdown` |
| **接口系统** | 用户管理系统 | 合同管理系统 |
| **数据范围** | 当前用户有权限的公司 | 所有合同公司（全量） |
| **查询方式** | 单次查询返回扁平数据 | 分级查询（一级+二级） |
| **使用场景** | 成本管理页面查询条件 | 用户权限配置 |
| **权限控制** | 后端根据 userBaseId 过滤 | 前端获取全量数据用于配置 |

## 二、DeptHierarchyTreeSelect 组件数据来源

### 接口信息

**文件位置**: `src/api/public/mdmWxDepartment.js`

```javascript
export function getLecUserCompanyRelByUserBaseId(data) {
  return request({
    url: `${BASE_API}/PlatformUmUserbaseinfo/getLecUserCompanyRelByUserBaseId`,
    method: 'post',
    data
  })
}
```

### 请求参数

```javascript
{
  userBaseId: string  // 当前登录用户的基础ID
}
```

### 返回数据结构

```javascript
{
  model: [
    {
      companyId: string,        // 公司ID
      companyName: string,      // 公司名称
      companyLv: number,        // 11=一级公司, 10=二级公司
      parentCompanyId: string   // 父公司ID（二级公司才有）
    }
  ]
}
```

### 数据特点

- ✅ **权限过滤**：只返回当前用户有权限访问的公司
- ✅ **用户维度**：不同用户看到的数据不同
- ✅ **扁平结构**：后端返回扁平数据，前端组装成树
- ✅ **单次查询**：一次请求获取所有数据

## 三、用户管理详情页面数据来源

### 接口信息

**文件位置**: `src/api/infolook/contractManagement.js` 或类似文件

```javascript
// 从代码中推断的接口调用
getContractOnlineDropdown({
  currentPage: 1,

  filterModel: {

  }

```

### 查询逻辑

**分两步查询**：

1. **查询一级公司**（包含集团本部）

```javascript
const levelOneRes = await getContractOnlineDropdown({
  filterModel: {
    contractCustomerType: '11',
    deepLevel: '1',
    requiresHeadquarters: true
  }
})
```

2. **并发查询每个一级公司的二级公司**

```javascript
await Promise.all(
  levelOneNodes.map(async (parent) => {
    const children = await fetchBusinessCompanyLevelTwo(
      parent.id, 
      parent.companyName
    )
    parent.children = children
  })

```

### 返回数据结构

```javascript
{
  model: {
    records: [
      {
        companyId: string,
        companyName: string,
        licenseName: string,        // 营业执照名称
        companyShortName: string,   // 公司简称
        uscc: string,               // 统一社会信用代码
        licenseNo: string,          // 营业执照号
        customerType: number,       // 11=一级, 10=二级（历史字段）
        companyLv: number          // 11=一级, 10=二级（新字段）
      }
    ]
  }
}
```

### 数据特点

- ✅ **全量数据**：返回所有合同公司，不过滤权限
- ✅ **分级查询**：需要分别查询一级和二级公司
- ✅ **合同维度**：数据来源于合同管理系统
- ✅ **配置用途**：用于配置用户可以访问哪些公司

## 五、关键差异对比

### 5.1 接口参数差异

**DeptHierarchyTreeSelect**：

```javascript

[
  { companyId: "1", companyName: "华东公司", companyLv: 11, parentCompanyId: null },
  { companyId: "2", companyName: "上海分公司", companyLv: 10, parentCompanyId: "1" }

```

**用户管理详情页面**：

```javascript
    records: [...]
  }

```

### 5.2 数据字段差异

| 字段 | DeptHierarchyTreeSelect | 用户管理详情页面 |
|-----|------------------------|----------------|
| 公司ID | companyId | companyId |
| 公司名称 | companyName | companyName / licenseName / companyShortName |
| 层级标识 | companyLv (11/10) | companyLv / customerType (11/10) |
| 父公司ID | parentCompanyId | 需手动传入 |
| 信用代码 | ❌ 无 | uscc / licenseNo |
| 父公司名称 | ❌ 无 | 需手动传入 |

### 5.3 树形结构构建差异

**DeptHierarchyTreeSelect**：

```javascript
// 后端返回扁平数据，前端根据 companyLv 和 parentCompanyId 构建树
```

## 六、本次需求调整的影响

### 对 DeptHierarchyTreeSelect 的影响

✅ **数据接口不需要修改**
- 接口已经包含 `companyLv` 和 `parentCompanyId`
- 数据结构已经支持一级/二级公司区分

⚠️ **需要新建级联组件**
- 创建 `DeptHierarchyCascadeSelect` 组件
- 复用现有的 `getLecUserCompanyRelByUserBaseId` 接口
- 前端将数据拆分为一级/二级公司选项

### 对用户管理详情页面的影响

❌ **不受影响**

**原因**：
1. 该页面用于配置权限，不是查询条件
2. 已经使用 `el-cascader` 级联选择器
3. 数据来源和结构不需要调整
4. 配置的权限会自动影响新组件的可选项

## 七、实施建议

### 短期方案

1. **保持用户管理详情页面不变**
   - 该页面已经是级联选择器
   - 数据来源无需调整

2. **新建级联组件替换成本管理页面**
   - 创建 `DeptHierarchyCascadeSelect` 组件
   - 使用现有的 `getLecUserCompanyRelByUserBaseId` 接口
   - 前端将数据拆分为一级/二级公司选项

3. **保持数据接口不变**
   - 后端接口已经支持层级区分
   - 前端组件负责数据转换和展示

### 长期优化

1. **统一数据源**
   - 考虑将合同系统和用户管理系统的公司数据统一
   - 建立公司主数据管理系统

2. **数据同步机制**
   - 合同系统新增公司时，自动同步到用户管理系统
   - 确保两个系统的公司数据一致性

## 八、总结

### 核心要点

1. **不是同一数据源**：DeptHierarchyTreeSelect 使用用户管理系统接口，用户管理详情页面使用合同管理系统接口
2. **数据用途不同**：前者用于查询条件筛选，后者用于权限配置
3. **数据通过权限关联**：用户管理页面配置的权限 → 用户权限数据库 → DeptHierarchyTreeSelect 组件
4. **本次需求不影响用户管理页面**：只需新建级联组件替换成本管理页面的查询组件
5. **可复用现有接口**：新组件可以复用 `getLecUserCompanyRelByUserBaseId` 接口

### 实施路径

```
新建 DeptHierarchyCascadeSelect 组件
    ↓
复用 getLecUserCompanyRelByUserBaseId 接口
    ↓
前端将数据拆分为一级/二级公司
    ↓
替换成本管理页面的查询组件
    ↓
用户管理详情页面保持不变
```
