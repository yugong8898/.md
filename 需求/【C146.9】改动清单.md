# 【C146.9】移动端-运单结算&运单付款-改动清单

## 需求概述

状态筛选改为**支持多选**，涉及：运单结算、运单付款、付款申请记录

---

## 一、通用规则（三个模块都遵循）

### 1. 多选交互规则

- **点击状态** → 切换选中/取消
- **"全部"和其他选项互斥** → 点"全部"清空其他，点其他清空"全部"
- **不选任何状态** → 后端查询全部数据

### 2. Tab 联动规则（白话版）

**核心逻辑**：一个 Tab 对应多个筛选状态（一对多关系）

例如：
- "待付款" Tab → 包含 3 个状态：待申请(100)、审核不通过(400)、支付失败(700)
- "付款中" Tab → 包含 3 个状态：待审核(200)、待付款(300)、付款中(500)

**场景1：点确定后跳到哪个 Tab？**
- **一律跳转到【全部】Tab**（不管选了什么状态）

**场景2：切换 Tab 后，筛选条件怎么变？**（重点）

需要动态判断和处理筛选项的选中状态：

- **有交集** → 只保留交集部分
  - 例：在"全部"Tab 选了 `[待申请, 待审核, 支付成功]`
  - 切到"待付款"Tab（包含 `[待申请, 审核不通过, 支付失败]`）
  - 取交集：只保留 `[待申请]`，其他自动取消勾选

- **没交集** → 重置为新 Tab 的默认状态
  - 例：在"已付款"Tab 选了 `[支付成功]`
  - 切到"待付款"Tab（包含 `[待申请, 审核不通过, 支付失败]`）
  - 无交集：清空筛选条件，重置为默认状态

**场景3：其他筛选条件（车牌号、客户名称等）**
- **切 Tab 时不动**，除非手动删除

---

### 3. Tab 联动技术实现参考

```javascript
// Tab 和状态映射表（付款申请记录示例）
const TAB_STATUS_MAP = {
  all: [100, 400, 700, 200, 300, 500, 600, 'processing'],
  pending: [100, 400, 700],   // 待付款
  paying: [200, 300, 500],    // 付款中
  paid: [600]                 // 已付款
}

// 高级筛选提交后的处理
onFilterSubmit() {
  // 不管选了什么状态，都跳转到【全部】Tab
  this.currentTab = 'all'
  this.fetchData()
}

// 手动切换 Tab 时的处理
onTabChange(newTab) {
  const currentSelected = this.filterForm.statusList  // 当前选中的状态
  const newTabStatus = TAB_STATUS_MAP[newTab]         // 新 Tab 包含的状态
  
  // 取交集
  const intersection = currentSelected.filter(s => newTabStatus.includes(s))
  
  if (intersection.length > 0) {
    // 有交集，只保留交集
    this.filterForm.statusList = intersection
  } else {
    // 无交集，重置为新 Tab 的默认状态
    this.filterForm.statusList = this.getDefaultStatus(newTab)
  }
  
  // 重新查询数据
  this.fetchData()
}
```

## 五、技术实施要点

### 5.1 筛选组件拆分

**背景**：原来运单付款和付款申请记录共用同一个筛选组件

**改动**：因为本次需求差异较大，建议拆分成两个独立的筛选弹窗

| 模块 | 筛选项 | 备注 |
|------|-------|------|
| 运单付款 | 支付状态（多选） | 保留其他原有筛选项 |
| 付款申请记录 | 付款状态（多选，10个枚举） | 删除"对账状态"和"支付状态" |

**拆分好处**：
- 避免条件判断混乱
- 各自维护映射关系和联动逻辑
- 降低代码耦合度

### 5.2 开发优先级

1. **先做运单结算**（逻辑相对简单，熟悉流程）
2. **再做运单付款**（拆分筛选组件）
3. **最后做付款申请记录**（最复杂，需改 Tab 结构）
