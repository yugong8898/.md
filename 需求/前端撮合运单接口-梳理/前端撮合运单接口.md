# 前端撮合运单接口梳理步骤

> 本文档记录对 `logisticsweb` 项目中运单相关接口（lmt-orch-user / lmt-orch-dispatcher / lmt-orch-shipper / lmt-orch-driver / lgi-orch-waybill / lgi-orch-wbquery）使用情况的完整梳理过程，供后续同类梳理复用。
>
> ⚠️ **梳理范围说明**：本次仅覆盖 `logisticsweb`（PC 端）项目。`wlyd-app-consigner`（托运人小程序/H5）、`wlyd-app-driver`（司机端）等其他项目对同一批接口的使用情况**未在本次梳理范围内**，需另行梳理。

---

## 一、梳理前置准备

### 1.1 确认接口清单来源

- 接口列表文件：`.md/需求/前端撮合运单接口-梳理/waybill-api-list.md`
- 路由列表文件：`.md/需求/前端撮合运单接口-梳理/route-list.md`
- 梳理目标项目：`logisticsweb`

### 1.2 确认服务与变量映射关系

在 `logisticsweb/src/config/index.js` 中确认各服务对应的 URL 变量名：

| 服务名 | 配置变量名 | 实际前缀 |
|------|---------|---------|
| lmt-orch-user | `lmtUser` | `/gateway/lmt-user` |
| lmt-orch-dispatcher | `lmtDispatcherUrl` | `/gateway/lmt-dispatcher` |
| lgi-orch-waybill | `lgiWaybillUrl` | `/gateway/lgi-waybill` |
| lgi-orch-wbquery | `lgiWbqueryUrl` / `lgi_wbquery_API` | `/gateway/lgi-wbquery` |
| lmt-orch-shipper | 无专用变量 | 直接硬编码或暂未使用 |
| lmt-orch-driver | 无专用变量 | 直接硬编码或暂未使用 |

> ⚠️ lmt-shipper / lmt-driver 没有独立配置变量，需额外做字符串扫描。

## 二、四轮验证步骤

### 第一轮：变量引用维度（从代码找接口）

**目的**：找出 `src/api` 目录下所有调用了目标服务变量的接口封装。

**命令**：
```bash
grep -rn "lmtUser\|lmtDispatcherUrl\|lgiWaybillUrl\|lgiWbqueryUrl\|lgi_wbquery_API" \
  logisticsweb/src/api/*.js \
  | grep "url:" | grep -v "^\s*//"
```

**产出**：带文件名 + 行号的完整调用清单，格式示例：
```
userBase.transport.js:577:    url: lmtUser + '/waybill-track/query-waybill-his-track',
```

**注意**：
- 结果包含非运单接口（如 `table-config`、`common-address`），需对照接口清单过滤
- 同一接口可能在多个文件中被封装

---

### 第二轮：函数溯源维度（从接口找调用方）

**目的**：对第一轮找到的每个接口，确认其封装函数是否被页面/组件/工具函数真实调用（排除"定义了但没用"的情况）。

**命令**：
```bash
grep -rn "函数名1\|函数名2\|函数名3..." \
  logisticsweb/src/views/ \
  | grep -v "^\s*//"
```

**产出**：每个函数的实际调用位置，格式示例：
```
views/userBase/vehicleMonitoringPositioning/index.vue:2400:  gpsDriverTrackList({ ... })
```

**注意**：
- 函数可能在 `src/utils/` 或 `src/components/` 中被中转调用，需一并检查
- `import` 语句本身不算调用，需确认有实际执行语句

---

### 第三轮：路径覆盖维度（查盲区）

**目的**：检查 `src/views`、`src/components`、`src/utils` 中是否有**绕过 api 文件、直接内联写接口 url** 的情况，防止漏掉不走 api 文件的调用。

**命令**：
```bash
grep -rn "lmtUser\|lmtDispatcherUrl\|lgiWaybillUrl\|lgiWbqueryUrl\|lgi_wbquery_API" \
  logisticsweb/src/views/ \
  logisticsweb/src/components/ \
  logisticsweb/src/utils/ \
  | grep "url:" | grep -v "^\s*//"
```

**产出**：有则列出，无则确认为空（exit code 1）。

---

### 第四轮：路径字符串硬编码维度（补盲区）

**目的**：检查是否有人跳过配置变量、直接把完整路径字符串（如 `/gateway/lmt-user/waybill/detail`）写在代码里。

**命令**：
```bash
# api 目录
grep -rn "lmt-user\|lmt-dispatcher\|lmt-shipper\|lmt-driver\|lgi-waybill\|lgi-wbquery" \
  logisticsweb/src/api/ \
  | grep "url:" | grep -v "^\s*//"

# views / components / utils 目录（分批，避免超时）
grep -rn "lmt-user\|lmt-dispatcher\|lmt-shipper\|lmt-driver\|lgi-waybill\|lgi-wbquery" \
  logisticsweb/src/components/ \
  logisticsweb/src/utils/ \
  | grep "url:" | grep -v "^\s*//"
```

**产出**：有则逐条比对接口清单；无则确认为空。

**注意**：
- views 目录体积大，全量扫描易超时，建议拆分为运单相关子目录单独扫描
- 硬编码结果需区分是否在接口清单范围内（本次发现的 `platform-fusion`、`enterprise-qualification` 等均不在范围内）

## 三、本次梳理结果汇总

### 3.1 四轮验证结论

| 验证轮次 | 维度 | 结论 |
|---------|-----|------|
| 第一轮 | 变量引用 | api 目录共找到 22 条 lmt/lgi 调用，其中 10 条属于运单接口范围 |
| 第二轮 | 函数溯源 | 10 条均有实际页面/组件调用，无"定义但未用"的情况 |
| 第三轮 | 路径覆盖 | views / components / utils 无绕过 api 文件的直接调用 |
| 第四轮 | 字符串硬编码 | api 目录有 12 条硬编码，全部为非运单接口；其余目录无 |

## 五、本次梳理局限性与待确认事项

### 5.1 梳理局限性

| 局限点 | 说明 | 建议处理 |
|------|-----|---------|
| 仅覆盖 logisticsweb | 其他项目（wlyd-app-consigner / 易达宝小程序等）对同一批接口的使用未梳理 | 需对各项目单独执行四轮验证 |
| 静态代码扫描 | 无法覆盖动态拼接路径（如运行时通过变量组装 url）的情况 | 人工走查关键业务流程页面做补充验证 |
| 注释代码未排除 | 部分接口可能存在于注释代码中，但扫描时已加 `grep -v "^\s*//"` 过滤，Vue 模板注释 `<!-- -->` 不在过滤范围 | 影响极小，暂可忽略 |
| 未验证接口是否仍在生产可用 | 接口存在调用不代表后端仍维护该接口，部分接口可能已废弃但前端未清理 | 需后端确认 130 条接口的现有维护状态 |

### 5.2 待确认事项

- [ ] `lmt-dispatcher`、`lmt-shipper`、`lmt-driver` 三个服务在 logisticsweb 上均未使用，是否属于**只在易达宝 APP/小程序使用、PC 端不接入**的设计？需与后端或产品确认，避免后续误判
- [ ] `waybill-api-list.md` 中"是否涉及网货运单"和"是否需要调整"两列目前均为空，需后续补充

## 六、后续梳理参考建议

1. **扫描命令中的变量名需每次对照 `config/index.js` 确认**，不同项目变量名可能不同
2. **lmt-shipper / lmt-driver 类无独立变量的服务**，只能通过第四轮字符串搜索覆盖
3. **views 目录全量扫描易超时**，建议优先扫描运单/结算/轨迹等业务相关子目录
4. **函数定义 ≠ 函数使用**，第二轮溯源是必要步骤，不可省略
5. **中间层调用需注意**：函数可能经由 `utils/` 或 `components/` 中转，不一定直接在 views 中调用
6. **注意新旧接口并存**：梳理时若发现同一功能有两个接口（新/旧），需记录切换逻辑，避免只改其中一个
7. **扩展到其他项目时**，重复执行四轮步骤，并将结果补充到 `waybill-api-list.md` 的"是否使用"列中，区分项目来源
